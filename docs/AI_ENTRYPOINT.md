# AI_ENTRYPOINT.md — AI 작업 시작점 (v2)

> Claude가 작업 시작 전 반드시 따라야 하는 절차.
> v2: 3-tier 자동 분기 + 티켓 자기완결. **단 이 프로젝트는 풀팀 모드라 micro 자동 흐름이 비활성**.

## Trigger
- 사용자가 작업 지시할 때
- 티켓 구현 요청받을 때
- 새 화면/기능 만들 때
- 자연어로 변경 요청할 때 ("로그인 버튼 오타 고쳐줘" 등)

## Action (v2 — 분기에 따라 다름)

### Step 0: 변경 사이즈 판별

| 흐름 | 진입 조건 |
|---|---|
| ~~micro~~ | **풀팀 모드라 비활성** — 모든 변경이 normal 이상 |
| normal | 1~10파일 + 한 도메인 |
| cross | 10파일↑ 또는 공통 모듈/디자인 토큰/auth |

### Step 0.5: 불확실성 인터뷰 (v2.1)

작업 결과가 달라질 수 있는 불확실성이 있으면 구현 전에 질문합니다.

- 질문은 한 번에 1~3개만
- 각 질문에는 추천값과 이유를 함께 제시
- 사용자가 "추천대로"라고 답하면 추천값 채택
- 일반 작업은 기본값으로 진행 가능하지만, auth/JWT/쿠키/API 계약/데이터 손실 위험은 답변 전까지 중단

질문하지 않아도 되는 경우:
- 티켓/SSOT/REQ에 기준이 있음
- 단순 문구, 파일명, 관례적 네이밍

### normal/cross 흐름

1. **티켓 확인 또는 생성**
   - 티켓이 있으면 그것 사용
   - 없으면 `ticket-writer` 에이전트로 v2 통합 양식 티켓 생성 (spec, plan, AC, 인라인 룰 통합)
   - REQ 기능 ID가 있으면 티켓 제목·추적자에 명시 (예: `LOGIN-FE-001`)

2. **컨텍스트 로드** (v2 — 가벼움)
   - **티켓 파일 1개만 필수**
   - 티켓의 "인라인 룰" 섹션이 충분한지 확인
   - 부족하면 그때만 도메인 문서 자율 추가 로드
   - UI 작업이면 `docs/design/DESIGN_SYSTEM.md`의 해당 컴포넌트 사양 확인

3. **영향 범위 보고** (자동 진행)
   - 어떤 파일을 수정/생성할 예정인지 목록 제시
   - 승인 받지 않고 보고만 (단, 사용자가 멈추라면 즉시)

4. **구현**
   - AI_AGENT_RULES 준수 / 인라인 룰 준수 / 최소 수정 원칙
   - semantic token만 사용, Park UI 베이스 + recipe

5. **검증 파이프라인 실행** (04 모듈 살뺀 버전)
   - type-check, lint
   - FF ×1 + 추가 1회까지
   - code-reviewer ×1 + 추가 1회까지
   - (cross + UI) Playwright

6. **회고 섹션 채우기**
   - 티켓 파일의 `## 회고` 섹션에 3~5줄 추가
   - ~~`docs/analysis/<ticket-id>_report.md`~~ — v2 폐기

## Forbidden (v2)

- 위 절차 건너뛰기
- 티켓 없이 코드 작성 (풀팀이라 예외 없음)
- 인라인 룰 무시
- v1처럼 `docs/analysis/` 파일 만들기
- 디자인 토큰/`panda.config.ts` 임의 변경
- auth/JWT/쿠키 코드를 가볍게 처리 (무조건 cross)
